# Git Flow 的核心分支
> Git Flow 模型定义了五个核心分支，它们各自有明确的职责：

## main (或 master)：
- 用途：这是主分支，永远代表已发布、可部署到生产环境的代码。
- 规则：这个分支上的所有提交都必须是稳定的。任何新功能或修复都不直接提交到这个分支，而是通过合并其他分支的方式进入。main 分支的每一次合并，都应该打上一个版本号标签（例如 v1.0.0），代表一个新的发布版本。

## develop：
- 用途：这是开发主线分支，所有新功能的开发都基于这个分支。它包含了所有已完成并准备好在下一个版本中发布的代码。
- 规则：所有新功能分支都从 develop 分叉出来，并在完成后合并回 develop。当一个版本的所有功能都开发完毕后，develop 分支会合并到 main 分支。

## feature (功能分支)：
- 命名：通常以 feat/ 开头，例如 feat/add-login-page。
- 用途：用于开发单个新功能。
- 规则：
  - 从 develop 分支创建。
  - 开发完成后，将该分支合并回 develop。
  - 完成后可以删除该分支。

## release (发布分支)：
- 命名：通常以 release/ 开头，例如 release/v1.1.0。
- 用途：用于准备一个新版本的发布。在这个分支上，可以进行最后的 Bug 修复、版本号更新等操作。
- 规则：
  - 当 develop 分支的代码已经足够稳定，可以准备发布时，从 develop 分支创建。
  - 所有针对该版本的最终修复只提交到这个分支。
  - 发布后，release 分支必须同时合并到 main 和 develop，以便 main 分支接收新版本，develop 分支接收 release 期间的 Bug 修复。

## hotfix (热修复分支)：
- 命名：通常以 hotfix/ 开头，例如 hotfix/fix-critical-bug。
- 用途：用于紧急修复生产环境 (main 分支) 上的 Bug。
- 规则：
  - 直接从 main 分支创建。
  - 修复完成后，必须同时合并到 main 和 develop。合并到 main 以立即修复生产环境，合并到 develop 以确保未来的开发版本也包含此修复。
