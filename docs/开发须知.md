# 开发须知

## 注意事项

在开始开发之前，请务必仔细阅读以下注意事项，以确保开发环境的统一和协作流程的顺畅。

- **Node.js 版本**: 本项目要求使用 `Node.js` v22 或更高版本。请在本地环境中确保 Node.js 版本符合要求。项目根目录下已配置 `.npmrc` 文件和 `package.json` 中的 `preinstall` 脚本，会在 `npm install` 时自动检查 Node.js 版本。

- **打开工作区**: 为了让项目中配置的 ESLint、Prettier 以及 VS Code 相关设置（如保存时自动格式化）能够正常生效，请务必使用 VS Code 打开项目根目录下的 `npm-audit.code-workspace` 工作区文件进行开发，而不是直接打开文件夹。

- **代码合并流程**: 任何代码的合并，无论是合并到 `main` 分支还是 `dev` 分支，都必须通过在 GitHub 上创建 Pull Request (PR) / Merge Request (MR) 的方式进行。禁止直接向这些受保护的分支推送（push）代码。

## Git 工作流规范

为了保持代码仓库的整洁、可追溯性以及自动化流程的顺利进行，我们约定了以下 Git 工作流规范。所有团队成员都应严格遵守。

### 1. 分支命名规范

所有分支的命名都应遵循 `类型/描述` 的格式。这样的命名方式可以清晰地看出每个分支的意图。

**格式示例:**

- `feat/user-login-module`
- `fix/fix-button-style-bug`

**分支类型说明:**

| 类型       | 描述                                 |
| :--------- | :----------------------------------- |
| `feat`     | 开发新功能 (feature)                 |
| `perf`     | 性能优化或功能优化 (performance)     |
| `refactor` | 代码重构，不改变现有功能             |
| `style`    | 调整代码格式，不涉及功能逻辑         |
| `docs`     | 添加、修改文档                       |
| `fix`      | 修复线上或测试环境的 bug             |
| `hotfix`   | 紧急修复线上的严重 bug               |
| `build`    | 影响构建系统或外部依赖的更改         |
| `chore`    | 其他杂项更改，如构建流程、辅助工具等 |

> **注意:** 本地会自动执行 `pre-commit` 钩子，检查分支名是否符合规范。如果不符合，你将无法提交代码。

### 2. 提交信息规范 (Commit Message)

我们采用 [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/) 规范来约束提交信息。规范的提交信息有助于我们快速浏览提交历史，并能自动生成版本更新日志 (Changelog)。

**格式:** `<类型>: <主题>`

**示例:**

- `feat: add user login endpoint`
- `fix: correct button alignment on homepage`

**类型说明:**

提交信息的类型与分支命名的类型完全一致，请参考上方的 **分支类型说明** 表格。

> **注意:** 本地会自动执行 `commit-msg` 钩子，检查提交信息是否符合规范。如果不符合，提交将会失败。
